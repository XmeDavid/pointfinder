# PointFinder Visual-System Audit - 2026-05-15

## 1. Executive Summary

Overall readiness: the visual system is ready to guide new work, but the implementation is not ready for a large visual refactor without a small enforcement layer first. Web has the strongest workspace direction and useful primitives, iOS has partial semantic token adoption, and Android has good feature coverage but duplicated color/status definitions.

Biggest sources of future drift:

- Status semantics are implemented locally in many screens instead of through shared `StatusBadge`/status-token components.
- Map markers are product-critical but still use duplicated hard-coded color maps and platform-specific marker implementations.
- Panel chrome is partially centralized on web through `GlassPanel`, but major overlays and native surfaces still rebuild surface/radius/elevation decisions inline.
- Empty, loading, error, offline, NFC, and sync states exist, but they are inconsistent components rather than shared product patterns.
- There is no visual harness/preview matrix to lock canonical states before refactoring.

Highest-leverage refactor targets:

1. Add canonical status tokens/components and a web visual harness.
2. Normalize web map overlays around `SurfacePanel`/`GlassPanel`/`InspectorPanel`.
3. Promote `BaseMarker`, `TeamMarker`, `SyncStatusBanner`, `NFCScanPrompt`, `EmptyState`, and readiness/status components into reusable platform equivalents.
4. Align cross-platform state naming and colors before touching screen layouts.

## 3. Finding Categories

### Tokens and colors

Web token anchors exist in `web-admin/src/index.css`, iOS anchors exist in `DesignTokens.swift`, and Android anchors exist in `Color.kt`. The problem is not absence of tokens; it is that feature code frequently bypasses them through raw palette classes, direct hex values, `Color.red`/`Color.blue`, or local status maps.

### Tailwind/web implementation

Web primitives are in place (`Button`, `Badge`, `Card`, `Input`, `Tabs`, `Tooltip`, etc.), but authenticated feature screens still contain raw palette Tailwind, repeated badge strings, direct overlay chrome, and local status logic. This conflicts most with `web-tailwind.md`, which says status and panel styling must be centralized.

### Component structure and reuse

The strongest reusable layer today is web primitives plus `GlassPanel`, `SlideDrawer`, `FloatingBar`, `EmptyState`, and map marker components. The missing layer is product components: canonical `StatusBadge`, `InspectorPanel`, `SurfacePanel`, marker variants, sync/NFC components, activity rows, review cards, and readiness items.

### Status semantics

The same concepts are colored in different places by local maps. Examples include game status (`setup` blue on web dashboard/topbar but amber/submitted on native setup), base progress (`checked_in` warning in one web inspector but blue elsewhere), and sync/offline (blue/red direct colors on native while web uses tokens in some places).

### Panels/cards/surfaces

Web command overlays are visually close to the desired map-first workspace, but panel chrome is copied in `ReviewOverlay`, `ResultsOverlay`, `BaseInspector`, and button-like stat tiles. Native uses many local `Surface`/`.background(.ultraThinMaterial)` patterns without shared product surface components.

### Map markers and overlays

Base and team markers are product components, but their status colors and shapes are implemented separately per platform. Web has `BaseMarkers`, `TeamMarkers`, and a legacy/common `MapMarkers`; iOS has `BaseAnnotationView` and `TeamLocationAnnotationView`; Android has bitmap marker factories plus map-screen-specific color functions.

### Forms and validation

Web has shared inputs and labels, but feature forms still mix native inputs/selects with local error text. Native platforms use platform controls correctly, but validation/error presentation is screen-local. Form primitives are less urgent than status/panel/marker work, but they should be normalized during setup/review passes.

### Empty/loading/error/offline states

All platforms have loading/error/empty/offline states, but they are inconsistent. Web has `EmptyState` and `Spinner`, yet many screens render local text blocks. iOS mixes `ContentUnavailableView`, private `emptyState`, alerts, and inline banners. Android uses local `Box + Text`, `CircularProgressIndicator`, and private sync/NFC states.

### Player/operator boundary

Mobile player surfaces mostly preserve the correct boundary: challenge titles are used instead of operator base names in several recently documented paths. The web auth/onboarding boundary needs product confirmation because web currently behaves as operator/admin auth only.

### Platform parity

Concept parity exists in feature coverage, but component parity does not. The same concepts are named and styled independently across platforms: sync pill/banner, readiness ring, activity event, leaderboard row, base marker, team marker, and status badge.

### Accessibility/localization/responsiveness

The codebase uses localization in many places and includes test IDs/accessibility identifiers. Risks remain around icon-only controls without accessible labels in web overlays, raw symbol/emoji use in `StageStrip`, and long translated text inside locally sized pills/buttons. No visual harness currently exercises long PT/DE strings.

## 8. Open Questions

- Should `setup` be blue/info, amber/caution, or gray/inactive across all platforms?
- Should `ended` be muted/gray or rejected/destructive red? Current native setup code sometimes treats it as rejected.
- Is web auth intentionally operator/admin-only, or should the web entry point also expose player join?
- Should `GlassPanel` keep `rounded-xl` as a map-overlay exception, or should web operational overlays standardize on `rounded-lg`?
- What is the canonical visual difference between `submitted`, `pending review`, and `sync pending` when both can occur in player flows?
- Should native operator live mode be map-first by default, or is list-first acceptable on small screens with a separate map tab?
- Should rank colors (gold/silver/bronze) become documented chart/result tokens?
- How much decorative motion is acceptable in NFC scan prompts during live field play, and should reduced-motion alternatives be explicit?
